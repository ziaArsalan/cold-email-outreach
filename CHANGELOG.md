### 2026-07-08 ‚Äî Fix: campaign Start double-submit (double-enqueue / double-send) ‚Äî ad hoc
- **Fixed:** Clicking **Start** twice on a campaign enqueued the batch twice and sent every email twice. Root cause: `campaignService.start()` only flipped the campaign `draft‚Üírunning` at the *end* (after the whole enqueue loop), so two near-simultaneous requests both passed the `status !== 'draft'` check and both ran the full enqueue. Now `start()` **atomically claims** the campaign `draft‚Üírunning` via `findOneAndUpdate({ _id, status:'draft' }, { status:'running' })` *before* enqueuing ‚Äî only one request wins; the loser gets a clear 400 "Campaign is already being started". Deliverability is still validated *before* the claim (a bad template leaves the campaign in draft), and if the enqueue phase throws mid-run the status rolls back to draft so a retry is safe (already-enqueued leads are excluded by the `status:'new'` filter + in-flight dedupe). The batch body was split into `enqueueBatch()` for the claim/rollback wrapper. Client: `campaignAction` now takes a synchronous `useRef` lock (drops a same-tick second click before React re-renders) and a `campaignActionId` busy state that **disables** the campaign's action buttons and shows "Starting‚Ä¶"/"Resuming‚Ä¶"; the start alert now notes sending runs in the background. `.btn-stop`/`.btn-ghost` gained `:disabled` styling.
- **Area:** both
- **QA:** Verified ‚Äî concurrency probe fired **two simultaneous** `start()` calls on one draft campaign: one enqueued 3, the other rejected, total queue items = **3 (one batch, not 6)**. Sequential double-start ‚Üí 2nd rejected "Only draft campaigns can be started". Regressions `SEQUENCES PASS` / `QUEUE PASS` / `ROTATION PASS` (on a cleaned DB ‚Äî earlier failures were stale `pending` queue litter the global `claimNext` was picking up, not this change); clean CRA build.
